# Gamebase — документация проекта

Веб-приложение для совместной игры в формате «общая комната по коду»: хост управляет партией и видит состояние на большом экране, игроки подключаются с телефонов по четырёхзначному коду комнаты. Сервер авторитативный: игровая логика выполняется на backend, клиенты получают состояние через Socket.IO.

На текущий момент в реестре игр реализован один режим — **«Угадай общее»** (`common_guess`).

## Запуск локально

**Бэкенд** (порт по умолчанию `3001`, переопределение: `PORT=3002 npm start`):

```bash
cd backend && npm install && npm start
```

Режим с автоперезапуском при изменении файлов:

```bash
cd backend && npm run dev
```

**Фронтенд** (порт `5173`, запросы к Socket.IO проксируются на backend):

```bash
cd frontend && npm install && npm run dev
```

Открыть в браузере: `http://localhost:5173`.

Продакшен-сборка фронта:

```bash
cd frontend && npm run build && npm run preview
```

Проверка живости API:

```bash
curl http://localhost:3001/health
```

Если при втором запуске сервера видите `EADDRINUSE`, порт уже занят предыдущим процессом — остановите его (`Ctrl+C`) или завершите процесс, слушающий тот же порт.

## Комнаты и роли

- Код комнаты — **4 цифры**, выдаётся при создании.
- **Ведущий (host)** — первый создатель комнаты; отдельное подключение по маршруту `/room/:code/host`, сокет получает состояние игры как **host state** (`getHostState`).
- **Игрок** — подключение `/room/:code/play`, роль `player`, участвует в игре и получает **player state** (`getPlayerState`).
- **Зритель (spectator)** — может подключиться без имени игрока; получает то же представление, что и ведущий (host state), без действий игрока.

Учитываются только участники с `role === 'player'` при подсчёте очков и обязательных ответов в фазах игры.

## События Socket.IO (обзор)

События задаются в `backend/src/server.js` и используются страницами хоста и игрока.

| Событие (клиент → сервер) | Назначение |
|---------------------------|------------|
| `host_create_room` | Создание комнаты, имя создателя |
| `host_rejoin` | Повторное подключение ведущего по коду и токену |
| `join_room` | Вход игрока или зрителя по коду комнаты |
| `start_game` | Старт игры ведущим: `gameType`, `options` |
| `host_game_action` | Действия хоста внутри игры (например `finalize_round`) |
| `player_action` | Действия игрока (слова в фазе сбора, `return_to_lobby` для создателя) |
| `SUBMIT_MATCH` | Ответ на сверку слова в «Угадай общее» |

| Событие (сервер → клиент) | Назначение |
|---------------------------|------------|
| `room_update` | Сериализованная комната (участники, статус) |
| `game_update` | Снимок состояния игры под роль получателя |
| `room_joined` / `join_room_failed` | Результат входа |
| `room_created` / `error_msg` | Создание комнаты и ошибки |
| `start_game_failed` | Ошибка старта игры |
| `answer_ack` | Подтверждение приёма слов |
| `submit_match_ack` | Подтверждение приёма сверки |
| `NEW_ROUND` | Рассылка при новом «микро-раунде» слова (broadcast) |
| `ROUND_RESULTS` | Итоги раунда сверки (broadcast) |
| `GAME_OVER` | Конец всей партии, таблица очков (broadcast) |

Канал комнаты в Socket.IO: `room:<код>` — используется для широковещательных обновлений.

## Игра «Угадай общее» (`common_guess`)

Пишите ассоциации к разным (иногда безумным!) темам и ищите совпадения с друзьями; чем больше совпадений — тем ближе победа.

Идея: по очереди задаётся **тема** (текст «вопроса» из банка). Игроки **тайно** вводят несколько слов-ассоциаций. Затем из **общего пула** всех введённых слов случайно показываются слова; игроки отмечают, есть ли у них на карточке совпадение с показанным словом. Совпадения нескольких игроков дают очки.

### Параметры старта (из UI ведущего)

Передаются в `start_game` внутри `options`:

- **`roundSeconds`** — время на ввод слов в начале каждого **макро-раунда** (в UI зафиксировано 60 с; на сервере допустим диапазон 15–600 с).
- **`totalRounds`** — число **макро-раундов** (1–5), каждому соответствует свой «вопрос» из банка `questions.json`.

### Банк вопросов

Файл `backend/questions.json` — массив строк. При старте из него **случайно** выбирается `totalRounds` неповторяющихся пунктов (с дозаполнением заглушкой, если в файле мало строк).

### Ограничения на слова игроков

- Не более **5** слов с одного игрока (`MAX_WORDS`).
- Нормализация: `normalizeWord` — обрезка пробелов, нижний регистр, первый «токен», обрезка пунктуации по краям, максимум 48 символов.
- Дубликаты одного игрока после нормализации отбрасываются.

### Жизненный цикл партии

1. **`collecting`** — макро-раунд сбора слов  
   - Таймер до `deadlineAt`.  
   - Игроки отправляют `player_action` с `{ words: [...] }` или одним `answer`.  
   - Хост может вызвать `host_game_action` с `{ type: 'finalize_round' }`, чтобы завершить сбор досрочно.  
   - По таймауту вызывается `_endCollectingPhase`.

2. После сбора строится **пул слов** `gamePool`: все нормализованные слова всех игроков перемешиваются; для каждого игрока создаются **слоты** `{ id: "<playerId>:w<i>", text }` и множество оставшихся id `remainingWordIds`.

3. Если пул пуст — игра сразу переходит в **`finished`**.

4. **`matching`** — серия проверок по одному случайному слову из пула:
   - Всем рассылается `NEW_ROUND` с `currentWord`, `roundId` (`roundSeq`), счётчиками пула и т.д.
   - Каждый игрок один раз за этот «ход» отправляет **`SUBMIT_MATCH`**:  
     `roundId`, `matched` (есть ли у выбранной карточки совпадение с показанным словом), `wordId` выбранного слота, **`wordText`** текста карточки (дублирует выбор для надёжности на сервере).
   - Пока не получены ответы от **всех** игроков, раунд не завершается.
   - Если кто-то не ответил за **120 с** (`ROUND_SUBMIT_MS`), недостающим подставляется `{ matched: false }`.

5. **`round_result`** — подведение итога микро-раунда (~3 с, `BETWEEN_ROUNDS_MS`), затем снова **`matching`** с новым словом или переход к п. 6.

6. Когда пул слов опустошён в рамках текущего макро-раунда — пауза **между макро-раундами** (~4.5 с, `BETWEEN_MACROS_MS`), фаза **`between_macros`**, затем новый **`collecting`** со следующим вопросом из `gameQuestions`.

7. После последнего макро-раунда — **`GAME_OVER`**, финальные очки и лидерборд.

### Механика сверки слова (matching)

Авторитетная логика в `CommonGuess._resolveRound`:

1. Для каждого игрока по его ответу определяется **карточка** (`slot`):
   - сначала по **`wordId`**, если id ещё в `remainingWordIds`;
   - иначе по **`wordText`** и нормализации, чтобы совпасть с одной из оставшихся карточек (защита от рассинхрона клиента).

2. Игроки с успешно сопоставленной карточкой и `matched === true` попадают в список валидных совпадений.

3. **Общий пул** обновляется так:
   - всегда удаляется **одно вхождение** слова раунда (то, что было вынуто случайно — позиция `savedPoolIdx` или иначе `currentWord`);
   - для каждого валидного совпадения дополнительно удаляется **одно вхождение текста карточки** игрока из пула и снимается соответствующий id с его «руки» (`remainingWordIds`).

Так поддерживаются **разные написания** одного и того же по смыслу слова (опечатки у разных игроков): игроки договариваются устно и оба отмечают «есть совпадение»; сервер не сравнивает строки карточки с `currentWord` буквально, а доверяет выбору слота и нормализованному тексту.

4. **Очки** за микро-раунд: если валидных совпадивших игроков **не меньше двух** (`N >= 2`), **каждому** из них начисляется **`+N`** очков (`pointsPerMatcher = N`). Если совпал только один игрок, очков за раунд нет, но снятие карточек и правки пула по правилам п. 3 всё равно выполняются для тех, кто корректно отправил совпадение.

5. Итог микро-раунда уходит в `ROUND_RESULTS` и в `lastRoundResults` в состоянии клиентов.

### Константы игры (сервер)

| Константа | Значение | Смысл |
|-----------|----------|--------|
| `MAX_WORDS` | 5 | Максимум слов от игрока за макро-раунд |
| `DEFAULT_ROUND_S` | 60 | Длительность сбора по умолчанию (сек) |
| `BETWEEN_ROUNDS_MS` | 3000 | Пауза между микро-раундами (словами) |
| `BETWEEN_MACROS_MS` | 4500 | Пауза между макро-раундами |
| `ROUND_SUBMIT_MS` | 120000 | Таймаут ожидания ответов на сверку |

## Расширение проекта новыми играми

1. Добавить класс игры в `backend/src/games/`, наследовать `BaseGame` и реализовать контракт (`onStart`, `onAction`, `onEnd`, при необходимости `onHostAction`, `getPlayerState` / `getHostState`).
2. Зарегистрировать тип в `backend/src/games/index.js` (`GAME_REGISTRY`).
3. На фронте добавить компонент игрока в `GameContainer.jsx` (`PLAYER_VIEWS`) и экран ведущего при необходимости.
4. Прокинуть выбор игры и настройки в модалке старта (`GameStartSetupModal.jsx` и вызов `start_game`).

## Зависимости и продакшен

- **CORS** на Express и Socket.IO включён для кросс-origin (в продакшене ограничьте `origin` под свой фронт).
- Для работы сайта и сокетов на одном хосте в dev используется **прокси Vite** для `/socket.io`; в проде нужен либо общий origin, либо явная настройка URL Socket.IO-клиента к API.
- Очки игроков дублируются в записи участника комнаты (`syncPlayerScores` в `RoomManager`) для отображения в лобби/после игры.
